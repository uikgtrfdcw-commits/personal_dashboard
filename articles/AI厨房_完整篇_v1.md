# AI 编程工具指南：用一个厨房比喻，讲清楚所有你需要知道的事

你用 AI 编程工具有一段时间了，一开始觉得很神奇，似乎无所不能。但用久了你发现，有些重复任务，每次都要重新交代背景，解释需求。

你开始想：能不能把这些重复的流程固定下来？

于是你撞上了一堆新概念，比如 Skill、MCP、Workflow、Rules、Memories。它们的功能看起来大同小异，到底有什么区别？我们可以通过一个比喻来说清楚。

假设你是餐厅老板，而 AI 是你雇佣的厨师。

这位厨师虽然聪明过人，什么菜都会做，但他第一天上班，对你的厨房还不熟悉，每道菜都要摸索很久。为了提高效率，你需要帮他把流程固化下来。

## 一、厨房里需要什么：六个核心概念

**首先是 Memories，也就是厨师的记忆。**他需要记住你的口味偏好，比如你不吃香菜、喜欢少盐，或者周二固定要做员工餐。这些并不是写在纸上的死规定，而是相处久了之后自然产生的默契。如果换一个厨师，你就得重新交代一遍。

Memories 在不同的软件里叫法各异。在 Windsurf 里它叫 Cascade Memories；Claude Code 里被称为 Auto Memory；GitHub Copilot 通过对话上下文和 Copilot Instructions 文件间接实现类似效果；Cursor 目前还没有独立的记忆功能，通常需要靠规则文件来间接实现。

**Rules 就像是贴在墙上的店规。**比如「所有菜必须少油」「出餐前必须检查摆盘」或者「不准用味精」。不管哪个厨师来都要遵守。Rules 和 Memories 的区别是：Memories 是厨师「记住」的，Rules 是厨房「规定」的。一个是软性偏好，一个是硬性约束。

同样的，Rules 也有很多名字。Windsurf 叫它 Cascade Rules；Cursor 里叫 Rules for AI 或 .cursorrules 文件；Claude Code 里一般写进 CLAUDE.md；GitHub Copilot 通过 .github/copilot-instructions.md 文件来配置；Cline 里叫 .clinerules 或 Custom Instructions。

**Workflow 指的是菜谱。**比如做红烧肉，第一步要焯水，第二步要炒糖色，第三步加酱油料酒，最后小火炖上 40 分钟。把这些写清楚并贴在灶台旁边，任何厨师照着做，都能做出味道统一的红烧肉。第一次做需要摸索，一旦写成菜谱，以后照着执行即可。

菜谱在 Windsurf 中直接叫 Workflows。在 Claude Code 中，你可以通过斜杠命令来实现类似的效果。GitHub Copilot 和 Cursor 没有内置的菜谱功能，但通常可以靠规则文件或提示词模板来替代。Cline 也是类似的思路。

**至于 MCP，则可以看作是厨房里的水电燃气管道。**厨师不需要自己发电或打井，灶台打开就有火，水龙头拧开就有水。你不需要关心水是从哪个水库来的，只需要知道水龙头在哪。而且，由于全世界的水龙头都有统一的口径，哪怕换个房子也能立刻接上。这就是「标准化」的意义所在。

MCP 是 Anthropic 提出的开放标准。目前 Windsurf、Cursor、Claude Code、Cline 都已经支持，GitHub Copilot 也在 2025 年加入了 MCP 支持。这也是少数几项在各大工具中称呼完全一致的概念。

**还有脚本，它相当于厨房里的预制菜和半成品。**葱姜蒜提前切好放在小碗里，常用的酱汁提前调好，高汤每天早上先熬一锅。点单来了直接取用，而不用从头切葱。如果删掉了脚本，就像每天把切好的葱扔掉再重新切，虽然能用，但非常浪费时间。

这个概念在所有编程工具里都存在，也就是你项目中保留下来的那些工具函数、配置文件或模板。

**最后是 Skill，指的是针对某个具体任务的打包方案。**比如「红烧肉」这个 Skill，包含了菜谱（Workflow）、调好的酱汁（脚本）、「做菜要少放盐」的备注（Rules）、「老板上次说肉要切大块」的记忆（Memories）。它将所有相关配置打包在一起，形成一个可复用的能力模块。

Skill 这个词主要在 Claude Code 社区流行，其他 AI 编程工具没有统一叫法，但本质是一样的：当记忆、规则、流程、脚本、外部连接都配置妥当，AI 在特定任务上就能开箱即用。

当以上配置全都齐全了，你走进餐厅，说一句：「老样子。」

菜就上了。

说到这里，你可能会问：那是不是要把所有任务都固化成流程？

不一定。红烧肉可以写菜谱，但「如何开发新菜式」这类问题，是没法写成菜谱的。流程固定的任务（比如数据抓取）可以把流程标准化，提高效率。但流程灵活的任务（比如项目规划、方案讨论），无法固化流程，只能固化工具——把食材备齐、灶台调好，至于今晚开发什么新花样，到时候再说。

这些概念之所以容易混淆，是因为它们在不同层面解决同一个问题：如何让 AI 避免每次都去「重新造轮子」。

其实你不需要一次性搞懂所有概念。先用起来，遇到重复性工作的时候想一想：这个步骤能不能固化下来？让 AI 帮你执行。慢慢的，整个体系就搭建完成了。

## 二、不同风格的厨房：主流工具一览

配齐了记忆、规则、菜谱、半成品和管道之后，下一个问题是：选哪种厨房？

AI 编程工具分两大类：IDE（集成开发环境）和 CLI（命令行工具）。IDE 就像一个装修好的厨房，灶台、冰箱、切菜板都摆在眼前，你能看到所有东西，用鼠标点点就能操作。CLI 则像一个没有厨房的餐厅，你只有一部对讲机，所有操作都靠打字下指令，厨师在后厨自己干。你看不到他怎么切菜、怎么调火，只能看到最后端出来的菜。对于编程新手来说，IDE 更直观友好；CLI 看起来原始，但老手用起来反而更快，因为省去了图形界面的操作开销，指令直达。

先说 IDE 类的。

**GitHub Copilot** 像老师傅身边的帮厨。你切菜的时候他看着你的手法，猜你下一刀要切哪里，提前帮你把菜摆好。活儿干得快，但前提是你自己得会做菜，他只是加速，不能替你决策。它是目前市占率最高的 AI 编程工具，嵌入在 VS Code 和 JetBrains 等主流编辑器里。

**Cursor** 像一个语音控制的智能厨房。你说「做一道红烧肉」，它会列出步骤让你确认，然后一步步执行。你对厨房有完全的掌控权，每一步都让你确认。它的 Rules 还有一个灵活版本叫 Notepads，相当于墙上的备忘板，你可以随时指着某张卡片说「按这个来」，不管换多少个厨师（新对话），抬头看一眼备忘板就能接上之前的工作。

**Windsurf** 像一个带管家的厨房。你说「做一道红烧肉」，管家不光做菜，还会主动检查冰箱里有没有肉、调料够不够，甚至提前把灶台预热好。单次做菜很省心。但管家靠记忆力工作，换班的时候（新对话）信息传递不如 Cursor 精确。

再说 CLI 类的。

**Claude Code** 像一个记性好、会聊天的对讲机。你说「做红烧肉」，它会先跟你讨论方案，然后自己去厨房执行。厨房门口有一本活页笔记本（CLAUDE.md），厨师每次进门会先翻一遍，下次做同样的菜不用重新交代。它是目前 CLI 领域最热门的工具，Skill 生态最成熟。

**Codex CLI** 像一个可以随时打断的对讲机。厨师在炒菜的时候你可以随时插嘴「少放点盐」，他会实时调整，不用等做完再重来。而且你可以同时给好几个厨师下单，每人做一道菜，并行工作。这是 OpenAI 出品的终端工具。

选哪个工具没有标准答案，取决于你的使用场景和习惯。但不管用哪个工具，那些核心概念（记忆、规则、菜谱、半成品、管道、打包方案）都是通用的。工具只是厨房的装修风格不同，做菜的道理是一样的。

## 三、厨师能多大程度独立工作：自动化的三个层次

配齐了工具、选好了厨房之后，还有一个关键问题：你需要在厨房里盯着吗？

这是很多人忽略的维度。同样是 AI 编程，自动化程度可以差出几个量级。

**第一层：盯着做。**

目前大多数 IDE 工具的工作方式。厨师每切一刀都回头看你一眼：「老板，这样切行吗？」你点头，他才继续。这就是所谓的 pair programming 模式——人和 AI 实时协作，每一步都需要人类确认。效率已经很高了，但你必须全程在场。

Windsurf 和 Cursor 都属于这一层。它们的界面设计——文件树、代码高亮、终端面板、diff 预览——这些都是给人看的。AI 每执行一步都要渲染到界面上，等你看懂了、点了批准，才能继续。灶台越漂亮，你越舍不得离开，越想盯着看。

**第二层：对讲机模式。**

CLI 工具天然更接近自动化。为什么？因为没有灶台，就没有需要「给人看」的东西。

Claude Code 执行完一个命令，直接读取文本输出，决定下一步，再执行。中间没有任何渲染、等待、确认的环节，除非它主动问你。你看不到厨师在干嘛，所以你不需要在场。你不需要在场，厨师就可以连续干活。

逻辑链很简单：有灶台 → 有东西给你看 → 你得在场 → AI 得等你 → 自动化受限。没灶台 → 没东西给你看 → 你不需要在场 → AI 自己跑 → 更高自动化。

灶台不是自动化的前提，灶台是自动化的阻碍。它的存在是为了服务人类的监督需求，而监督本身就是自动化的反义词。

**第三层：交钥匙。**

2026 年 2 月，OpenAI 发布了 Codex Agent。这是一种全新的模式：你用自然语言描述需求，AI 在云端沙箱中自主规划、写代码、启动应用、测试、发现 bug 后自己修复，循环迭代直到满意，最后通知你验收。

用厨房比喻：你早上走进餐厅，说「今晚有十桌客人，八道菜，菜单在这里」，然后去忙别的事。四个小时后回来，菜已经做好了。厨师甚至自己试了味道，觉得咸了又调整过一遍。

这之所以能实现，有一个关键前提：编程的核心操作是读写文本文件和执行命令，AI 可以直接操作，不需要「看」屏幕。视觉只在最后验证环节用一下——启动应用、截个屏确认界面正常——整个工作量的 90% 以上是纯文本操作。

这和另一类全自动工具（比如 Manus）形成了鲜明对比。Manus 操作的是通用桌面——浏览器、表格、各种软件——它的每一步都需要截屏、用视觉模型理解屏幕内容、再决定下一步操作。这就像一个蒙着眼的厨师，每动一下都要拍照看看手里拿的是什么、锅在哪里、火开了没有。视觉处理极其昂贵：一张截图消耗的 token，够你和 AI 深度对话好几分钟。全程走视觉路线，四个小时的成本可能要几百美元。

所以目前真正实用的「交钥匙」模式，集中在编程领域——因为代码是纯文本，AI 可以直接读写，成本低、精度高。通用桌面操作的全自动化，还要等视觉处理成本大幅下降之后才能普及。

未来的趋势很可能是三层融合：简单任务交钥匙，日常开发对讲机模式，关键决策拉你回来在灶台前确认。就像一个成熟的厨师，日常出餐不用你管，但遇到 VIP 客人的特殊要求，会主动来问你。

## 补充：关于索引功能

一家小餐厅，厨师一眼就能看到所有食材。但如果你开的是一家大酒店呢？几百种食材分散在不同的冷库、仓库、调料间里，不可能全部搬到灶台上。

好厨师不会试图把所有东西都摆在眼前。他只需要知道仓库的布局：肉类在冷柜 A，蔬菜在架子 B，调料在柜子 C。做红烧肉的时候，去拿需要的东西就行。

AI 也是一样。它的「灶台」大小是有限的（上下文窗口），不可能同时看到所有东西。所以上面提到的记忆、规则、菜谱、半成品，并不是一股脑全塞给 AI，而是让 AI 先拿到一张「仓库地图」。这在技术上被称为 RAG，也就是检索增强生成。

无论是 Cursor 的代码库索引、Windsurf 的自动关联拉取，还是 Claude Code 让模型自己执行搜索命令，都是在做同样的事情。
