# 你逛知乎多久了，如何看待知乎如今的氛围？

知乎能学 GitHub 吗？
我常常会想：如果知乎这样的写作平台像GitHub 一样运营，会发生什么？
乍一看这很荒谬。一个是讨论网络热点和治国理政的地方，一个是程序员协作写代码的地方。但仔细思考，两者做的事情本质上是一样的：把分散在不同人脑子里的知识汇聚成某种可用的东西。
区别在于汇聚的方式。
在知乎上，如果有人写了一篇关于经济学的文章，而你发现了一个逻辑漏洞，你的选择是：写一篇新文章反驳他，或者在评论区吵一架。无论哪种，结果都是信息的进一步碎片化。你的洞见和他的洞见永远是两个独立的东西，读者得自己在脑子里做整合。
在 GitHub 上，情况完全不同。如果你发现别人代码里有 bug，你不需要另起炉灶发一个新项目，而是可以直接提交一个修复（Pull Request）。如果被接受，你的贡献就融入了原来的项目。信息不再是碎片化的堆堆砌，而是像开源软件一样，集合众人的智慧不断优化用户体验。
还有版本控制。现在的文章一旦修改，旧版本就消失了。你无法看到一个观点是如何演变的。但系统会保留所有历史记录。你可以看到一个洞见如何从模糊的直觉，经过反复修改，最终变得精确。这比任何头衔都更能建立用户信任。
如果评论区变成 Issue Tracker （可以理解成一个问题清单，程序员用它来记录所有待修的 bug）会怎样？每条评论不再是情绪宣泄，而是一个待解决的逻辑问题。讨论会变得有目标感：要么解决这个问题，要么承认它超出了当前人类知识的边界。
这一切听起来很美好。但细想其实行不通。
首先，代码有客观标准。能跑就是对的，跑不了就是 bug。但一个历史观点或社会学洞见有没有 bug？谁说了算？结果就是“合并冲突”永远解决不了，最后演变成中文维基百科那种无休止的编辑拉锯战。
还有虚荣心的问题。GitHub 能运转，是因为程序员这个行业已经接受了“代码开源”这个前提。但写作者有极强的署名欲和领地意识。让一个大V接受读者的 Pull Request，把别人的逻辑加进自己的文章？心理上绝无可能。
最致命的是认知负荷。大多数人上网是为了消遣，不是为了协作。GitHub 那套东西对普通用户来说太复杂，门槛太高了。
不过，沿着这个思想实验走下去，不难发现一个残酷的事实：人类大多数所谓“创见”，不过是对现有知识结构的微小增量：一点拓展、一点跨领域嫁接、一点通俗化改写，小到根本不值得单独开一个仓库。但在社交平台上，这些微小增量都要伪装成石破天惊的洞见——标题要惊悚，观点要极端，不然没人点进来。流量需要新鲜感的幻觉，而不是真正的新知。
知识进步需要的其实恰恰相反：承认自己的贡献只是拼图的一小块，然后甘愿把它交出去，融进一个更大的东西里。GitHub 之所以能运转，是因为程序员愿意做这件反人性的事。
程序员群体之所以能客观评价自己的贡献，不是因为他们更高尚，而是因为代码会说话——你的贡献有没有用，有多大用，运行一下就知道了。这种残酷的客观性压制住了人性中那点傲慢。但在观点和见解的领域，没有编译器替你裁决，每个人都可以坚称自己的想法值得独立存在。
所以我的结论是：在大众社交平台上全面 GitHub 化不现实。但在学术协作、垂直技术社区、小型研究团队这些地方，这几乎是必然趋势。
区别在于你想要什么。有些人写东西是为了被看见，有些人是为了把事情搞清楚。这两种冲动都是真实的，只是它们指向完全不同的产品。大多数人一辈子都在第一种里打转，这也没什么错。但如果你发现自己反复被那些低水平重复的噪音弄得心烦，发现自己真正想要的是和别人一起把某个问题想清楚，那也许你该换个地方待着了。
