# 计算机学生大二，只喜欢算法相关，对繁琐的工程没有兴趣以后该怎么发展?

- 日期: 2025-12-20
- 赞同: 0
- 评论: 0

---

这个问题简直是在演我当年的内心戏。
十年前，多少计算机专业的学生（包括我）都是这么想的：觉得写Java、配环境、调CSS是搬砖的苦力活，只有推导公式、设计模型结构才是智慧的结晶，才有那种“上帝视角”的掌控感。
那时候，这种想法或许还能苟一苟。但在大模型已经把行业底裤都卷掉的今天，这种思维极其危险。中流211的大二学生，如果不抓紧时间对“工程”这两个字祛魅，等到大四找工作，大概率会撞得头破血流。
先说个反直觉的结论：
工业界根本就没有脱离工程的纯算法岗位。
那些让学生觉得“没有思考乐趣”的工程，通常是因为还没摸到真正的门槛。
很多同学在学校里接触的工程项目，确实挺无聊的。无非就是SSM框架搭一搭，CRUD（增删改查）写一写，遇到问题百度一下报错信息，把代码粘过来跑通就算完事。这种过程，确实像是在拼图，毫无逻辑美感可言。
但这不叫工程，这叫
堆砌
。
真正的工程挑战，和解一道LeetCode Hard题目的快感是一模一样的，甚至更强。
举个现在最火的例子。大家都知道RAG（检索增强生成）是大模型落地的标配。表面上看，它就是把文档切碎了存进向量数据库，然后搜出来给大模型看。
但真到了生产环境，问题全是算法级的：
几亿条数据，怎么在50毫秒内搜出最相关的Top 5？这里面涉及HNSW索引的底层原理，涉及内存与精度的数学博弈。
用户的Query千奇百怪，怎么做语义改写？这需要对NLP算法有极深的理解。
高并发来了，显存爆了，怎么做推理服务的调度优化？
解决这些问题，需要的不是“查资料”的能力，而是极强的
计算机体系结构
功底和
算法思维
。
如果觉得工程无聊，大概率是因为书读得还不够深。这里必须安利那本被无数架构师奉为圭臬的神书——
DDIA
（Designing Data-Intensive Applications，数据密集型应用系统设计）。去啃啃这本书，就会发现，分布式系统的每一个设计决策，本质上都是数学概率和逻辑的权衡。看完之后，再看那些繁琐的配置和架构，眼里看到的就不再是枯燥的代码，而是精妙的逻辑流动。
马上进2026年了，行业里最显著的变化就是：
传统的算法工程师正在消失，取而代之的是AI工程师（AI Engineer）。
以前，懂个XGBoost，会跑个ResNet，就能去面算法岗。那时候模型还得自己训，参数还得自己调，大家戏称“调参侠”。
现在呢？基座模型太强了。
绝大多数中小厂，甚至大厂的非核心部门，根本不需要人去从头训练模型。大家都在用开源的Llama、Qwen，或者直接调OpenAI的接口。
这就导致了一个很尴尬的局面：单纯会推导公式、会写模型Demo的人，价值被极度稀释。因为模型本身已经不仅仅是代码了，它成了基础设施。
现在最稀缺的是什么人？是
懂模型边界，且能把模型塞进复杂业务系统里的人
。
比如，怎么让大模型稳定地输出JSON格式？怎么通过Prompt Engineering（提示词工程）甚至微调，让模型学会写SQL？怎么设计一套Agent框架，让模型能自己调用搜索工具、计算器？
这些工作，表面看是工程，是写接口，是写逻辑控制流，但内核是对AI能力的深刻理解。
在这个背景下，你一个中流211的大二学生，如果非要死磕“纯算法研究”（Research Scientist），路会走得非常窄。那种岗位现在基本是留给顶级名校博士发Paper用的，神仙打架，凡人遭殃。
更务实的策略是：
用算法思维做工程，用工程手段落地算法。
现在正是大二，补基础的黄金期。千万别只盯着机器学习那几本书看，那是上层建筑。
要去啃
CSAPP
（深入理解计算机系统）。这本书是卡内基梅隆的镇校之宝。它会讲清楚代码是怎么从文本变成机器指令的，内存是怎么管理的。读通了它，写代码时那种“盲人摸象”的焦虑感就会消失，取而代之的是一种透视底层的掌控感。
另外，一定要去拥抱
Linux
。别整天在Windows IDE里点点点。去黑框框里敲命令，去写Makefile，去搞懂Docker容器。这些看似繁琐的“杂活”，是通往高薪的必经之路。因为现在的算法模型，全是跑在Linux集群上的。连环境都配不明白，怎么去训练千卡的大模型？
还要打破一个幻想：以为做算法就是每天喝着咖啡思考数学模型，做工程就是苦哈哈地搬砖。
真相是，无论做到多高级的算法专家，
80%的时间都在做Dirty Work。
数据科学家大部分时间在洗数据，处理那些缺胳膊少腿的脏数据。
算法专家大部分时间在分析Bad Case，盯着日志看模型为什么又说胡话了。
架构师大部分时间在写文档、开会、跟产品经理撕逼。
没有哪份工作是纯粹享受思考乐趣的。如果有，那只能是数学家，还得是不用操心经费的那种。
最后，给未来的职业发展定个调子。
别把自己框死在“算法”或者“开发”的标签里。
未来的几年，最值钱的是
Problem Solver
——解决复杂问题的人。
能自己清洗数据，自己训练（或微调）模型，自己写后端接口，自己部署上线，把一个想法变成一个可用的产品。这种“全栈AI工程师”，是接下来几年市场疯抢的对象。
试着去GitHub上找个热门的开源大模型，比如Llama 3或者Mistral，自己动手部署一下。试着去修改它的推理代码，试着给它挂载一个知识库。在这个过程中，会遇到无数的报错，无数的坑。
把这些坑填平的过程，就是成长的过程。
当有一天，面对一个报错，不再是慌张地去百度，而是能大概猜到底层哪里出了问题，甚至能直接去翻源码找到答案时，就会发现，原来所谓的“繁琐工程”，其实充满了硬核的解谜乐趣。
别怕脏，别怕累。在AI时代，动手能力才是最大的护城河。
