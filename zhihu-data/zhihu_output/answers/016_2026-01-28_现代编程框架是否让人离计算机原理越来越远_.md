# 现代编程框架是否让人离计算机原理越来越远？

- 日期: 2026-01-28
- 赞同: 0
- 评论: 0
- 链接: https://www.zhihu.com/question/1964654155080660003/answer/1999813631278338145

---

这是一个非常有意思的话题。
我自己面试过不少人，从刚毕业的名校硕士到工作七八年的资深开发都有。这几年确实有一个非常明显的趋势，就是大家对于框架的熟练度越来越高，甚至可以说是出神入化。给我一个业务需求，不管是做后端的高并发接口，还是搞深度学习的模型训练，现在的年轻人能在极短的时间内搭出一套像模像样的东西。
这当然是好事。技术的初衷本身就是降低门槛，提升生产力。
但是，这中间存在一个巨大的隐形成本，很多人在这个过程中不知不觉就交了学费，甚至断送了职业生涯的上限。
我经常在面试里问一个很经典的问题，尤其是针对那些简历上写着精通各类微服务框架或者深度学习框架的候选人。我会问，当你发起一个网络请求，或者在显存里加载一个模型的时候，操作系统到底发生了什么？数据是怎么从网卡到内存，又是怎么从内存到CPU缓存的？在这个过程中，如果发生了性能抖动，你觉得大概率是哪一层出了问题？
绝大多数时候，我得到的回答是一阵沉默，或者是试图用框架里的配置参数来解释底层原理。
这就触及到了你提到的核心问题。我们现在的开发环境，实际上是一个高度封装的舒适区。
在这个舒适区里，内存似乎是无限的，I/O似乎是瞬间完成的，并发似乎是由上帝在管理的。你调用一个函数，它就返回结果，至于中间经历了多少次上下文切换，产生了多少次缺页中断，或者触发了多少次垃圾回收，框架都帮你屏蔽了。
这看起来很美好。对于初级工程师，或者仅仅是完成业务逻辑的堆砌，这完全足够了。
可问题在于，真实世界的系统从来都不是运行在理想环境里的。
当业务量上来之后，当QPS翻了十倍百倍之后，或者是当你需要把一个大模型压缩到边缘设备上运行的时候，那些被框架屏蔽掉的
计算机原理
，就会像幽灵一样跑出来找你麻烦。
这时候你就会发现，框架提供的抽象层开始漏水了。你引以为傲的快速开发能力，在面对CPU 100%却找不到原因、内存泄漏却抓不到对象、网络延迟忽高忽低这些问题时，变得毫无用处。
我一直持有一个观点，那就是框架并没有让人离计算机原理越来越远，而是让人
误以为
自己可以离计算机原理越来越远。
这中间的区别很大。
真正的高手，是利用框架来减少重复劳动，但在他们的脑海里，依然清晰地运行着底层的逻辑图谱。他们知道这行代码写下去，对应的汇编指令大概是什么量级，知道这个对象创建出来，会占用多大的堆空间。
现在的很多年轻开发者，缺少的正是这种透视能力。
我们来拆解一下，为什么这种黑盒模式会削弱掌控力。
首先是对于
性能边界
的无知。
在算法领域，这点尤其明显。现在是2026年，大模型已经是基础设施了。大家都在用PyTorch，用各种高级的封装库。很多同学调参是一把好手，各种SOTA的模型信手拈来。但是如果你让他去优化一下推理速度，或者写一个自定义的CUDA算子，很多人就傻眼了。
他们习惯了Python层面的操作，习惯了自动求导，习惯了动态图。他们不理解GPU的内存层级结构，不知道什么是SM（流多处理器），不知道Warp Divergence（线程束分歧）会对性能造成多大的杀伤。
结果就是，模型跑起来了，但是算力利用率极低。公司买了昂贵的H100集群，被他们跑出了A100甚至更低的效果。这在商业上是巨大的浪费。
如果不懂底层原理，你就永远不知道性能的瓶颈在哪里。你只能像盲人摸象一样，去试各种参数，去碰运气。这种解决复杂问题的能力，是极其低效的。
其次是对于
系统稳定性
的误判。
现代框架为了易用性，往往会做很多激进的默认配置。比如某些Web框架，默认的线程池策略或者是连接复用机制，在低并发下表现很好，但是一旦流量突增，立刻就会导致系统雪崩。
一个不懂TCP/IP协议栈的人，是看不懂连接池为什么要这么设计的。他不知道TIME_WAIT状态意味着什么，不知道滑动窗口是如何控制流量的。当线上系统出现大量的连接超时，他除了重启服务，没有任何排查手段。
这种人，只能做顺风局的开发。一旦遇到逆风局，遇到真正的技术攻坚战，立马就会现原形。
我在团队里经常强调，不要做API的搬运工。
如果你只是熟练调用API，那么你的可替代性太强了。现在的AI代码助手，比如GitHub Copilot或者更高级的Devin，它们调用API的能力比你强一万倍。它们不会拼写错误，记得住所有参数。
人类工程师的核心竞争力，必须建立在对
系统本质
的理解上。
这种理解，是AI目前还很难完全替代的。因为AI是基于概率生成代码，它学到的是模式。而你需要具备的是逻辑推理和物理层面的直觉。
比如，当你看到一个服务的响应时间呈现出周期性的毛刺，你的直觉应该告诉你，这可能跟Java的GC机制有关，或者是操作系统的Page Cache回写策略有关。这种直觉，建立在你对计算机体系结构深刻的理解之上。
那么，回到你的问题，这种趋势会不会影响未来的技术创新？
我的答案是肯定的，而且影响深远。
创新往往发生在边界被打破的时候。如果你连边界在哪里都不知道，怎么去打破它？
很多突破性的技术，都是因为开发者不再满足于现有的抽象层，决定向下挖掘而诞生的。
Redis的作者Antirez如果不深入理解内存布局和I/O多路复用，绝对写不出Redis。Docker的诞生，是因为有人深刻理解了Linux内核的Namespace和Cgroup机制。
如果我们的一代代工程师，都只停留在应用层的框架上，满足于CRUD的业务逻辑，那么基础软件的创新就会枯竭。我们将很难再看到像Kubernetes、Kafka或者TensorFlow这样由底层原理驱动的系统级创新。
这其实是一个很危险的信号。
现在国内的互联网行业，虽然应用层非常繁荣，但是在基础架构、数据库内核、编译器、操作系统这些硬核领域，我们依然缺乏足够多的人才。
很大程度上，就是因为我们的教育体系和职场环境，太过于强调
立竿见影
的产出。
大家都在赶工期，都在拼上线速度。能用现成的轮子，绝对不自己造。能用高级语言解决，绝对不看底层实现。
久而久之，大家就丧失了向下探索的意愿和能力。
我这几年带新人，发现一个很有趣的现象。
那些能够走得远、成长得快的人，往往都有一个共性，就是
好奇心过剩
。
他们不满足于代码跑通了。他们会去翻看框架的源码，会去研究为什么这个函数比那个函数快。他们会在遇到Bug时，兴奋地去抓包，去分析堆栈，而不是简单地把错误日志扔到搜索引擎里找现成的解决方案。
这种钻研精神，是抵抗技术黑盒化的唯一解药。
当然，我并不是鼓励大家去造轮子，或者在写业务代码时非要用汇编。那是矫枉过正。
框架和底层原理，从来都不是对立的。
你可以把框架看作是你的武器。你要熟悉武器的每一个性能参数，知道它的射程，知道它的后坐力，知道它在什么环境下会卡壳。
而计算机原理，就是你的内功。
内功深厚的人，飞花摘叶皆可伤人。哪怕给他一个很烂的框架，他也能通过精妙的配置和魔改，发挥出最大的效能。反之，内功不行的人，给他一把屠龙刀，他也只能用来切菜，甚至可能伤到自己。
对于现在的年轻开发者，我的建议其实很务实。
你在工作的前三年，可以追求广度，熟悉各种框架，快速完成任务。这能让你在公司站稳脚跟。
但是到了三五年这个坎，如果你想突破瓶颈，想从高级工程师晋升到专家，或者架构师，你就必须回过头来，去补计算机原理这门课。
这不是让你去背书。而是结合你这几年的工程经验，去反刍那些枯燥的理论。
你会发现，当年在学校里觉得晦涩难懂的《深入理解计算机系统》，或者是操作系统里的进程调度算法，原来在实际工作中每天都在发生。
你会明白，为什么MySQL的索引要用B+树，这跟磁盘的物理读写特性有什么关系。你会明白，为什么高并发系统要尽量使用无锁编程，CAS指令到底比互斥锁轻量在哪里。
这些知识，才是你职业护城河的基石。
现在技术的迭代速度比十年前快了不知道多少倍。新的框架每年都在出，新的概念层出不穷。
如果你只追逐框架，你永远在疲于奔命。今天学React，明天学Vue，后天又出了个什么新玩意。你永远是新手。
但是计算机原理，那些底层的机制，几十年来其实没有太大的变化。
冯·诺依曼架构还在那里，操作系统的虚拟内存机制还在那里，TCP/IP协议栈还在那里。
掌握了这些不变的东西，你再去看那些千变万化的框架，就会有一种
一览众山小
的感觉。你会发现，所有的框架，无非就是对这些底层原理的不同排列组合和权衡取舍罢了。
比如，不管是Node.js的事件循环，还是Go语言的协程，本质上都是在解决如何利用有限的线程资源去处理海量并发连接的问题。如果你懂了I/O模型和线程调度，这两种技术在你眼里就没有本质的区别，你也能很快上手任何一门新的类似的语言。
所以，回答你的问题。
框架确实把底层细节封装起来了，但这不代表我们应该远离它。相反，正是因为封装得太好了，我们才更需要主动去揭开那层盖子，去看看下面的齿轮是怎么咬合的。
这不仅是为了解决复杂的线上故障，更是为了保持我们作为工程师的尊严和价值。
在这个AI辅助编程日益强大的时代，纯粹的代码生成工作正在贬值。
未来的顶尖工程师，一定是那些能够驾驭AI，同时又具备深厚底层功力的人。他们知道让AI生成什么样的代码是安全的，高效的。他们能够一眼看出AI生成的代码中潜在的性能隐患，并指出修改的方向。
这需要极高的技术品味，而技术品味，源自于对原理的深刻理解。
接下来的十年，是属于
全栈原理型
工程师的时代。这里的全栈，不是指你会写前端也会写后端，而是指你既懂上层的业务架构，又懂底层的硬件原理，你能打通这两者之间的任督二脉。
从今天开始，在用框架干活的同时，多问几个为什么。
当你在Python里写下
import torch
的时候，想想它底层的C++扩展是怎么加载的。当你在Java里写下
new Object()
的时候，想想它在堆内存里是怎么布局的，对象头里存了什么。当你发送一条消息到Kafka的时候，想想它是怎么通过零拷贝技术落盘的。
这些思考，虽然不会立刻体现在你的KPI里，但它们会一点一滴地重塑你的技术思维。
