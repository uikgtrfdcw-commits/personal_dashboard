# 大语言模型（LLM）推理的两个核心阶段：Prefill（预填充） 与 Decode（解码）是什么？

---

这个问题问到了点子上，也暴露了很多人对大模型推理优化的一个误区。
回头看这两三年，大模型推理架构的演进其实就干了一件事：在这个算力依然昂贵、显存带宽依然是瓶颈的时代，如何把GPU的每一滴性能都榨干。
Prefill 是一次性大计算，Decode 是高频小读取，这个总结非常精准，基本上把 LLM 推理的两个极端特质概括完了。至于 Chunked Prefill 到底怎么让 Decode 提前开始，这确实是一个反直觉的技术点。很多人第一反应是：这不符合因果律啊，我不把前面的 Prompt 读完，怎么能生成后面的 Token？
其实这里有一个概念上的错位。Chunked Prefill 并不是让当前这条请求的 Decode 跑到它的 Prefill 前面去，因果律目前还是物理学大厦的基石，没人能打破。它的核心魔法在于调度和流水线掩盖。
咱们直接看底层的计算逻辑和显卡的工作状态。。
Prefill 阶段，模型拿到你输入的一大段 Prompt，它需要一次性计算出所有的 Key-Value Cache（KV Cache）。这个过程是计算密集型的。你想想，输入一万个 Token，Attention 层的计算量是和长度的平方成正比的，这是一个巨大的矩阵乘法运算。这时候，GPU 的 Tensor Core 跑得火星直冒，算力利用率（MFU）非常高，基本上能把显卡的算力打满。这时候最大的瓶颈通常是计算单元够不够快。
Decode 阶段呢，情况完全反转了。模型每次只生成一个 Token。为了生成这一个 Token，它需要把之前所有 Token 的 KV Cache 从显存（HBM）里读出来，和当前的 Token 做计算。这是一个典型的访存密集型任务。计算量极小，但是数据搬运量极大。这时候 GPU 的计算单元大部分时间在 摸鱼，等着显存把数据搬过来。这就是为什么我们总说 Decode 阶段受限于显存带宽（Memory Bound）。
好了，传统的推理系统，也就是几年前的那种 naive 做法，通常采用的是 FCFS（先来先服务）策略。
想象一个场景：
用户 A 发来了一个 100k 长度的超长文档，要求总结。
用户 B 发来了一个只有 10 个 Token 的问题：今天天气怎么样？
在老式的系统里，GPU 必须先一口气把用户 A 的 100k Token 全部 Prefill 完。这个过程可能持续几秒钟甚至更久。在这期间，GPU 的计算单元确实在忙，但是显存带宽可能并没有吃满（取决于具体的算子实现）。最要命的是，用户 B 的请求虽然简单，但必须排队等着。
这就导致了一个严重的问题：Head-of-Line Blocking（队头阻塞）。用户 B 的 Time to First Token（TTFT，首字延迟）直接爆炸。
这时候，Chunked Prefill 进场了。
它的核心思想是：谁规定 Prefill 必须一口气跑完？
我们可以把用户 A 的那 100k Token，切成很多个小块，比如每块 1024 个 Token，我们叫它 Chunk。
这时候，GPU 的调度器手里就有了一堆任务卡片：
用户 A 的 Chunk 1
用户 A 的 Chunk 2
...
用户 A 的 Chunk 100
用户 B 的 Decode 步骤 1
用户 B 的 Decode 步骤 2
Chunked Prefill 的真正威力在于 混合调度。
当系统发现显卡里同时有正在做 Prefill 的任务（用户 A）和正在做 Decode 的任务（用户 C、用户 D...）时，它不再是非黑即白地二选一。
它会把用户 A 的一个 Chunk（比如 512 个 token）拿出来，和用户 C、D 的 Decode 步骤打包在一起执行。
你可能会问，Prefill 是矩阵乘法，Decode 是矩阵向量乘法，这俩怎么打包？
在底层算子层面，这其实就是把不同的计算核（Kernel）塞到同一个 Batch 里，或者在微观的时间片上进行交错。更先进的框架，比如我们现在常用的 vLLM 或者各个大厂自研的推理引擎，甚至能在 Attention 算子内部做融合。
这么做直接解决了两个问题，也回答了你关于 Decode 提前开始 的疑问。
第一，它消除了队头阻塞，让 别人的 Decode 提前开始。
这是最直观的解释。对于那个问天气的用户 B，系统不需要等用户 A 的 100k Token 全部算完。系统算完用户 A 的 Chunk 1 之后，就可以立刻插入用户 B 的 Decode 步骤。在宏观时间尺度上，用户 B 感觉自己的请求几乎是立刻开始生成的，完全没有被那个 100k 的大任务卡住。所以这里的 Decode 提前开始，通常指的是 系统层面其他并发请求的 Decode 阶段，不需要等待大任务的 Prefill 完成。
第二，它填补了计算和访存的空隙，让 GPU 这种吞金兽别闲着。
这是一个更硬核的视角。
我们刚才说了，Prefill 算力利用率高，Decode 显存带宽利用率高。
如果你只做 Prefill，显存带宽可能还没跑满。
如果你只做 Decode，计算单元在闲置。
Chunked Prefill 允许我们构造一个 混合 Batch。在这个 Batch 里，既有计算密集的 Chunked Prefill 任务，又有访存密集的 Decode 任务。
这就像是俄罗斯方块，把计算负载和访存负载拼在了一起。当 GPU 在等 Decode 的数据从显存搬运过来的时候，它正好可以用闲置的 Tensor Core 去算 Prefill 部分的矩阵乘法。
结果就是，显卡的 Compute 和 Memory Bandwidth 都能被推向极限。
这里要纠正一个常见的误解。Chunked Prefill 不仅仅是为了降低延迟，它本质上也是在提升吞吐。
在今天，我们看推理性能，早就不是看单卡的延迟了，那是 2023 年大家还在玩 Demo 时候的指标。现在我们看的是 tokens/s/dollar（每美元每秒生成的 token 数）。
如果不做 Chunked Prefill，为了保证即时性，我们往往需要预留一部分显卡专门处理小流量，或者限制 Max Batch Size，这都是对算力的浪费。有了 Chunked Prefill，我们可以大胆地把 Batch Size 开大，因为就算进来一个超长任务，我也可以把它切碎了慢慢嚼，不会把整个系统噎死。
再深挖一点，Chunked Prefill 到底是怎么切的？
这可不是随便切切就行。切得太小，Kernel Launch 的开销就上来了，GPU 一直在上下文切换，效率反而低。切得太大，又回到了阻塞的老路。
现在的调度器通常是动态的。它会根据当前显存的剩余容量、Pending 的请求数量，动态决定下一个时间步执行多少个 Prefill Token 和多少个 Decode Token。
这就涉及到 KV Cache 的管理。你把 Prefill 切开了，意味着用户 A 的中间状态（KV Cache）必须在显存里驻留，等待下一个 Chunk 进来继续计算。这也就是为什么 PagedAttention 这种技术在当年一出来就成了标配。因为只有非连续的、分页的显存管理，才能灵活地支持这种切碎的计算过程，防止显存碎片化导致 OOM（内存溢出）。
还有一个比较前沿的视角，是关于 Prompt Caching 的。
如果你的 Prefill 是基于很多重复的 System Prompt 或者 RAG（检索增强生成）检索到的公共文档。Chunked Prefill 结合 Radix Attention 或者类似的前缀树缓存技术，效果更炸裂。
我们不需要等用户的请求完全结束才开始计算。只要用户输了一部分，或者系统预判到某些公共前缀，我们就可以在后台利用 Chunked Prefill 的机制，利用 Decode 阶段的空闲算力，偷偷地把这些前缀先算好存起来。
等到用户真正的请求发出的那一瞬间，我们发现前面 90% 的 Prefill 工作已经做完了，直接从 Cache 里读 KV，只需要计算最后那一点点新增的 Token，然后立马进入 Decode 阶段。
这也是一种 Decode 提前开始，而且是针对当前请求的真正的加速。这在代码补全场景下用得非常多。程序员在打字的时候，IDE 里的模型其实已经在疯狂地 Prefill 当前文件的上下文了，等你停手的一瞬间，补全结果直接弹出来，感觉就像是本地计算一样快。
当然，Chunked Prefill 也不是没有代价的。
最大的坑在于 Attention 的计算逻辑。标准的 FlashAttention 算法在处理切分后的 Query 时，需要特别小心。因为 Attention 是全局的，后面的 Token 需要看到前面的所有 Token。当你计算 Chunk 2 的时候，你必须把 Chunk 1 产生的 KV Cache 加载进来。
这意味着，虽然我们把 Prefill 打散了，但是总的读取量其实是增加了的。因为每次计算新的 Chunk，都要去读旧的 KV。如果切得太细，重复读取 KV Cache 带来的显存带宽压力，可能会抵消掉混合调度带来的收益。
所以算法工程师在这里要做极其精细的权衡（Trade-off）。我们要找一个 Sweet Spot，平衡队头阻塞延迟和总吞吐量。
现在的SOTA（State of the Art）框架，比如 DeepSeek 或者 Meta 开源的那些底层库，在处理这个问题时，通常会用到一种叫 Split-K 或者类似的分块并行策略，配合 Chunked Prefill。
总结一下。
Prefill 是为了理解过去，计算密集，容易阻塞。
Decode 是为了预测未来，访存密集，算力闲置。
Chunked Prefill 并不是扭转因果律的黑科技，它本质上是一种细粒度的计算调度策略。
它把 大块的计算任务（Prefill） 拆解，填充到 频繁的小块访存任务（Decode） 的空隙中。
它让 Decode 提前开始，主要体现在两个维度：
系统维度：让排在后面的短任务不用等前面的长任务算完，直接插队运行。
流水线维度：通过混合计算，掩盖了访存延迟，让整个系统的响应速度变快，变相让所有人的 Decode 都更丝滑了。
这就是为什么到现在，任何一个上生产环境的大模型推理服务，如果不支持 Chunked Prefill，基本上就别谈什么商业竞争力了。算力成本那么贵，让 GPU 里的计算单元哪怕闲置一毫秒，都是对股东的不负责任。
再说一个实际的案例，咱们做搜索业务的时候，经常遇到这种情况。
用户的 Query 很短，但是我们需要从数据库里捞出几十篇相关的文档作为 Context 塞给模型，这就构成了典型的 Long Context 输入。
以前没上 Chunked Prefill 的时候，只要有一个用户搜了一个生僻词，触发了大规模文档召回，整个推理实例的延迟波形图就跟心电图一样，猛地跳一下。那几百毫秒里，其他用户的请求全部卡死。运维群里报警就没停过，P99 延迟惨不忍睹。
后来切了支持 Chunked Prefill 的引擎。你会发现，虽然那个长 Context 请求的处理总时长（Latency）并没有显著减少（甚至因为切分可能有微小的增加），但是整个集群的吞吐量（QPS）上去了，最重要的是，P99 延迟那条线，平得像刚铺好的柏油路。
这就是技术细节带来的体感差异。用户不会知道什么是 Prefill，什么是 Decode，他们只知道，这玩意儿回消息快不快。而我们要做的，就是在底层把这些复杂的计算像玩俄罗斯方块一样，严丝合缝地塞进 GPU 的每一个时钟周期里。
如果你以后面试，遇到有人问你这个问题，千万别只背概念。你得告诉他，这背后的本质是算力与带宽的博弈，是延迟与吞吐的权衡。这才是工程师该有的视角。
在实际落地中，还有一个非常隐蔽的坑，很多人容易忽视，那就是 KV Cache 的显存占用波动。
当你开启 Chunked Prefill 之后，系统中会同时存在多个处于 半熟 状态的请求。以前是做完一个扔一个，现在是大家一起慢慢做。这意味着同一时间，你需要保留更多请求的中间状态 KV Cache。
这对显存容量提出了更高的要求。如果你的显存本身就捉襟见肘，开了 Chunked Prefill 反而可能导致更频繁的 Cache Eviction（缓存驱逐），把好不容易算好的 KV 丢掉，下次又要重算，那性能反而是倒退的。
所以，这又回到了那个经典的话题：没有银弹。
所有的系统设计，最后都是在做资源置换。Chunked Prefill 是用更复杂的调度逻辑和潜在的显存压力，去换取更平滑的延迟表现和更高的计算单元利用率。
现在硬件也在进化。新一代的 GPU，比如 Blackwell 架构之后的卡，在硬件层面就开始支持更灵活的 Tensor Core 调度，甚至有的 NPU 架构开始尝试把 Prefill 和 Decode 的物理单元解耦（Disaggregated Serving），这又是另一个大的话题了。
也就是把 Prefill 专门交给算力强的节点做，把 KV Cache 传给带宽大的节点做 Decode。这也是一种极致的 Chunking，只不过是跨节点的 Chunking。
但万变不离其宗，核心逻辑依然没变：别让计算等数据，别让数据堵计算。
只要你能把这两句话悟透，不管未来出什么新名词，Chunked Prefill 也好，Split-Fuse 也好，你都能一眼看穿它的底裤。
