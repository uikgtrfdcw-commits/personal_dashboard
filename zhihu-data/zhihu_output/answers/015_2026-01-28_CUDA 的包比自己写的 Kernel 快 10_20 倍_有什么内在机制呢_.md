# CUDA 的包比自己写的 Kernel 快 10~20 倍，有什么内在机制呢？

这事儿真不赖你代码写得烂，纯粹是两个维度的较量。
我也年轻过，当年拿着《算法导论》里的伪代码用 CUDA C 复现，跑完一看耗时，心态直接崩了。别说慢 10 倍，某些极端的尺寸下慢 50 倍都正常。
你以为你在跟 cuFFT 比算法复杂度，其实人家在跟你比汇编级的微操和对 GPU 架构的极限压榨。
这 10 倍的差距，咱们拆解开看，每一层都能吃掉你一大截性能。
最要命的根本不是计算，是访存。
FFT 这个算法天生对 GPU 不友好。它的访存模式极其跳跃。第一级蝴蝶运算可能访问相邻元素，到了后面几级，读写的跨度能大到跨越数千个字节。你用 CUDA C 直接写，大概率是直接操作 Global Memory。
问题来了，GPU 的显存控制器是按 32 字节或 128 字节的粒度来存取的。你代码里写的是读一个 float，硬件层面实际上拉了一整块数据下来。如果你的线程束里的 32 个线程，读的地址是不连续的，硬件就得发射 32 次内存请求才能凑齐这波数据。带宽利用率瞬间跌到几十分之一。
cuFFT 怎么干？它根本不会让这种大跨度访问直接打在显存上。它会强行把数据先搬到 Shared Memory里。
这里还有个坑。Shared Memory 有 32 个 Bank。如果你代码写得不讲究，让多个线程同时去搞同一个 Bank 的不同地址，这就叫 Bank Conflict。一旦冲突，并行变串行，延迟直接起飞。cuFFT 会精细计算每一个数据的落点，甚至故意在数组里塞空数据，就是为了错开访问，保证 32 个线程能在一个周期内把数据拿完。
光这一手访存优化，就能拉开 3 到 5 倍的差距。
再说计算密度的压榨。
教科书教你 Radix-2算法，一次处理两个点。好写，但在 GPU 上这是低效的。
GPU 也是有流水线延迟的，从发出指令到拿到结果需要等待。掩盖这个延迟的最好办法就是多干活。cuFFT 内部大量使用 Radix-4、Radix-8 甚至更高。一个线程一次读进 4 个、8 个数。这样做有两个硬核好处：第一，减少了读写总次数；第二，增加了寄存器内的计算量。
你算两个点，寄存器趴窝等着；人家算八个点，ALU忙得冒烟，刚才读数据的延迟就在计算过程中被掩盖过去了。这叫 Instruction Level Parallelism。你用 CUDA C 写，编译器未必敢给你做这么激进的循环展开。
接着聊聊寄存器和指令。
很多人觉得 C 语言已经够底层了，但在高性能库开发者眼里，C 还是太抽象。
NVCC 编译器也是人写的，它的寄存器分配策略是通用的，偏保守。而 FFT 这种计算极其密集的算子，对寄存器极其敏感。多用一个寄存器，可能就会导致 SM上能跑的 Block 数量减少，Occupa占用率直接腰斩。
cuFFT 的核心代码往往是混着 SASS写的。工程师会人肉排布每一条指令，精确控制变量的生命周期。哪里该复用寄存器，哪里该插入控制指令防止流水线停顿，全都算死了。
还有那个取模运算。写 FFT 肯定要算下标，少不了用 
%
 和 
/
。在 GPU 上，整数除法和取模是超级昂贵的操作，耗时几十个时钟周期。cuFFT 会用位运算技巧，或者利用硬件的特殊指令来绕开除法。你自己写代码如果不注意这一点，几千次取模操作就能把性能拖垮。
更高级的还有 Warp Shuffle。
对于短序列的 FFT，cuFFT 甚至都不走 Shared Memory。它直接利用 
__shfl_xor_sync
 这种指令，让数据在寄存器之间飞。连片上内存的读写开销都省了。这种黑科技，常规的算法描述里根本不会提，但在硬件执行上就是降维打击。
最后一点，cuFFT 并不是一个 Kernel。
它是一个庞大的 Kernel 库加上一套复杂的启发式搜索逻辑。当你调用执行函数时，它其实是在根据你的输入长度 N、显卡架构（是 A100 还是 4090）、显存余量，去查表或者现场搜索最优解。
针对质数长度，它有 Bluestein 算法；针对 2 的幂次，它有 Stockham 算法。你用一套通用的几十行代码去硬抗所有情况，在工程上是不现实的。
所以，慢 10 倍是正常的。
你写的 Kernel 关注的是数学逻辑的正确性。
cuFFT 关注的是指令吞吐、显存带宽饱和度、流水线延迟掩盖。
想缩短这个差距，就不能只看算法书，得去看 PTX 指令集文档，去啃 GPU 架构白皮书。这不到 20 行的代码背后，其实是计算机体系结构的厚重壁垒。
