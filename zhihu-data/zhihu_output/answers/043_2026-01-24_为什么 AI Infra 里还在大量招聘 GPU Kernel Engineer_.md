# 为什么 AI Infra 里还在大量招聘 GPU Kernel Engineer？

- 日期: 2026-01-24
- 赞同: 0
- 评论: 0
- 链接: https://www.zhihu.com/question/1998102573333385508/answer/1998357017128093045

---

很多人都有这个疑问，既然NVIDIA提供了cuBLAS，开源社区有了CUTLASS，Meta和OpenAI都在推Triton，甚至PyTorch 2.0的Inductor都在做自动算子生成，为什么各家大厂还要养一堆高薪的Kernel Engineer去手撸CUDA？
这笔账如果只从软件功能的完备性算，确实很难算通。但如果从算力成本、硬件演进速度以及业务场景的极端性这三个维度去算，招聘Kernel Engineer不仅是必要的，甚至是目前性价比最高的投资。
你提到的cuBLAS和CUTLASS确实成熟，但它们的成熟是建立在
通用性
基础上的。通用往往意味着妥协。NVIDIA的官方库必须照顾所有可能的Shape，保证在几百种不同的矩阵尺寸下都能跑出及格的分数。这就好比一件均码的衣服，谁都能穿，但谁穿都不合身。
在实际的大模型训练和推理中，Shape是非常固定的，也是非常极端的。比如在大语言模型的解码阶段，Batch Size可能很小，Seq Length很长，或者反过来。这种长尾的Shape刚好落在官方库优化曲线的平缓区，甚至死角区。
这时候，只要你针对这个特定的Shape，把Block Size调一调，把Thread Layout改一改，甚至只是精细控制一下Shared Memory的加载流水线，性能就能提升30%甚至更多。
在一个万卡集群上，30%的性能提升意味着什么？
意味着原本需要跑30天的训练任务，现在20天就能跑完。如果租用H100集群的成本是每小时几千美元，这省下来的钱足够养一直这种Kernel团队好几年。这就是为什么大厂算完账后，会毫不犹豫地招人。这里的逻辑不是技术上的
能不能
，而是商业上的
值不值
。
更深层的问题在于
Memory Wall
，也就是存储墙。
现在的GPU算力增长速度远远超过了显存带宽的增长速度。A100到H100，算力翻了好几倍，但带宽并没有同比例增长。这就导致了一个现象：大多数时候，GPU的核心计算单元（Tensor Core）都在空转，等待数据从HBM（高带宽内存）搬运到SRAM（片上共享内存）。
cuBLAS这种库，主要解决的是矩阵乘法本身的计算效率。但在大模型里，真正的瓶颈往往不在矩阵乘法本身，而在矩阵乘法前后的那些操作：Cast、Transpose、Add、LayerNorm、Activation。这些操作计算量极小，但需要把庞大的数据搬进搬出。
如果你用PyTorch原生的写法，这些操作会启动很多个Kernel，每个Kernel都要读写一遍显存。这就是在浪费带宽。
Kernel Engineer的核心工作之一就是
算子融合
。把Conv、Add、Relu这一连串操作融合成一个Kernel，数据一旦读进寄存器或Shared Memory，就一口气算完再写回。这能把带宽需求降到最低。虽然CUTLASS现在也支持Epilogue融合，但它能覆盖的组合是有限的。业务模型里的魔改千奇百怪，今天搞个SwiGLU，明天搞个Rotary Embedding，后天又来个MoE的Top-K路由机制，这些新出来的结构，官方库根本来不及适配。
等你指望NVIDIA更新cuBLAS支持这些新算子，黄花菜都凉了。想要业务上线快、性能好，只能自己下场写。
你可能会问，编译器不能做这个事吗？TVM、XLA、Triton不就是干这个的吗？
确实，编译器的上限很高，但下限也很低，而且很不稳定。
自动代码生成工具目前面临的最大问题是
搜索空间太大
和
成本模型不准
。对于一个复杂的融合算子，有多少种Tiling的方式？有多少种流水线并行的策略？有多少种寄存器分配的方案？这是一个天文数字。编译器往往只能在一个简化的模型上做启发式搜索。
这就导致在一些关键路径上，编译器生成的代码总是差那么一口气。它可能不知道这里多用两个寄存器会导致Occupancy下降，从而无法掩盖访存延迟；它可能不知道在某种特定的硬件架构上，某种特殊的访存指令组合能触发更高的带宽利用率。
而且，调试编译器生成的代码是一场噩梦。当性能不达标时，你很难直接去改生成的PTX代码，只能去调编译器的参数，这往往变成了玄学调参。
相比之下，手写Kernel虽然开发周期长，但你是完全掌控硬件的。你知道数据在每一级缓存是怎么流动的，你知道每一个Thread在干什么。
举个真实的场景案例。
在做量化推理（Quantization）时，比如W8A8或者W4A16。你需要把权重解压缩，然后做矩阵乘法，还要处理各种Scaling Factor。这种混合精度的计算，加上极其特殊的内存布局（为了适应Tensor Core的输入要求），现有的编译器生成的代码往往效率极低。
这时候，一个资深的Kernel Engineer会利用PTX指令集里的异步拷贝指令（cp.async），配合精细设计的软件流水线（Software Pipeline），手动管理Shared Memory的Buffer，实现计算和访存的完美重叠。这种极致的优化，目前任何自动化工具都还做不到。
关于协作模式，你的理解有一点偏差。真实的工作状态并不是每个人分一块固定的蛋糕，守着那几个算子做到天荒地老。
通常的协作模式是
Profiling驱动
。
团队里会有专门负责Workload Analysis的人，或者大家轮流做。他们会在真实的业务模型上跑Profiling，用Nsight Systems或者Nsight Compute去抓数据。分析出来的结果通常是一个帕累托分布：也许有5个Kernel占据了80%的运行时间。
这5个Kernel就是全村的希望，是必须要攻克的山头。
这时候，任务会分发下去。通常是经验最丰富的人去攻坚最难的那个，比如核心的Attention或者是超大维度的MoE计算。资历浅一点的同学负责一些Element-wise的融合算子，或者负责把写好的Kernel集成到框架里，做单元测试，做数值对齐。
并没有所谓的流水线分工说有人专门写代码，有人专门做测试。Kernel Engineer必须具备全链路的能力。你写了Kernel，你就得负责它的正确性，你就得负责它的性能回归。因为GPU编程极其容易出错，Race Condition、Bank Conflict，这些bug只有写代码的人最清楚可能出在哪里。
而且，协作中很大一部分精力花在与算法团队的沟通上。
很多时候，算法团队提出的结构在硬件上是非常不友好的。比如某种稀疏注意力的Mask生成方式，可能导致极其严重的Warp Divergence（线程束分化）。Kernel Engineer需要懂算法，能看懂论文，然后告诉算法同学：你这个结构虽然理论上FLOPs少了，但在GPU上跑起来反而会变慢，建议改成另一种数学上等价但在硬件上更友好的实现。
这种
算法-硬件协同设计
（Co-design）的能力，是这个岗位的高级形态。
这个岗位需要多少人？
从绝对数量上，它永远不会像Java后端或者前端那么多。一个几千人的算法团队，可能只需要几十个顶尖的Kernel Engineer就能支撑。
但是，只要以下三个条件存在，这个岗位就不会饱和，反而是极度稀缺：
硬件架构在不断更新（Volta -> Ampere -> Hopper -> Blackwell）。
模型架构在不断演进（RNN -> Transformer -> Mamba/RWKV -> ?）。
摩尔定律失效，算力增长赶不上模型参数增长。
每当NVIDIA发布新一代架构，原本优化的代码可能就需要推倒重来。Hopper架构引入了TMA（Tensor Memory Accelerator）和WGMMA，如果你不用新的指令集，就发挥不出新硬件的性能。这就意味着每一代硬件都需要人去重新适配和压榨。
每当算法界搞出新花样，比如最近流行的长上下文，对显存管理和算子效率提出了全新的要求，以前的FlashAttention V2可能就不够用了，需要开发V3、V4。
所以，这个行业不存在人力饱和，只存在
能力饱和
。只会调cuBLAS接口的人早就饱和了，甚至被淘汰了。但能读懂SASS（汇编），能手写Pipeline，能看懂PTX ISA文档，能针对特定架构做极致榨干的人，永远不够。
护城河在哪里？
很多人觉得写CUDA就是C++加点API，这是巨大的误解。
这条路真正的护城河是对
计算机体系结构
的深刻理解。
你要懂GPU的存储层级，L1、L2、HBM的带宽 latency 都是多少？Cache Line是多大？Coalesced Access（合并访问）怎么做？
你要懂线程调度，Warp Scheduler是怎么工作的？什么样的指令序列能最大化指令流水线的吞吐？
你要懂并行的思维，如何把一个数学问题拆解成成千上万个微小的任务，并且这些任务之间还要尽量减少同步和通信。
其次是对
数值计算
的敏感度。
FP16、BF16、FP32甚至FP8，在不同场景下的精度损失是多少？如何处理累加器的溢出？为什么有时候改了加法的顺序结果就不对了？
最后才是
工程能力
。
如何管理复杂的Template代码？如何设计好的API？如何做自动化测试？
至于你提到的多机通信、Cluster Topology，那更多是Distributed System Engineer或者Network Engineer的范畴，虽然有交集（比如算子内通信融合），但通常是另一拨人在做。不过，顶级的GPU专家通常也懂通信，因为到了节点内通信（NVLink）这一层，计算和通信已经很难分开了。
如何布局与应对变迁
如果你想走这条路，不要把自己绑定在某一个特定的库（比如CUTLASS）或者某一个特定的模型（比如Llama）上。
你要做的是向下扎根。
去学体系结构，去搞清楚GPU到底是怎么跑起来的。去啃NVIDIA的Whitepaper，去理解SIMT（单指令多线程）模型的本质。
同时，要保持对编译技术的关注。虽然我说手写还有空间，但长远看，Triton这类DSL（领域特定语言）是趋势。它试图在手写汇编和傻瓜式调用之间找平衡。学习Triton的源码，理解它在IR（中间表示）层面做了哪些优化，是连接手工优化和自动化的桥梁。
在这个领域，真正的专家不是写代码最快的人，而是那个对着Nsight图表看一眼，就能指出“这里Shared Memory Bank Conflict了”或者“这里寄存器溢出导致Spill了”的人。
这种洞察力，是用无数个通宵对着机器调试换来的，是AI暂时学不会，也是开源库无法替代的。这才是你最坚实的护城河。
总结来说，AI Infra里大量招聘GPU Kernel Engineer，是因为在千亿参数的模型面前，在数亿美元的集群面前，哪怕1%的性能提升，都值得投入一个人力去死磕。这不是一种低效的重复造轮子，而是在摩尔定律放缓的时代，人类为了延续算力增长神话所做的最硬核的努力。
深入技术细节：为什么CUTLASS等开源库在实际生产中总是差点意思
要真正理解这个岗位，我们不能只停留在商业逻辑上，必须下沉到技术实现的毛细血管里。
CUTLASS，确实是NVIDIA推出的神兵利器，特别是CUTLASS 3.x引入了CuTe Layout之后，表达能力强了很多。但你要知道，CUTLASS的设计目标是提供一套
模板元编程的积木
。它给你提供了很好的抽象，比如MmaAtom、TiledMma、CopyAtom。
但是，积木是死的，搭积木的人是活的。
在实际业务中，我们经常遇到这样的情况：为了追求极致的端到端延迟，我们需要把Attention算子里的Softmax操作融合进前面的GEMM（矩阵乘法）里，或者把后面的GEMM和前面的GeLU融合。这种跨算子的深度融合，往往打破了标准GEMM的边界。
CUTLASS的标准Epilogue融合虽然强大，但它通常假设Epilogue是Element-wise（逐元素）的操作。如果你想在Epilogue里做一些Reduce（归约）操作，或者需要访问相邻行的数据，CUTLASS原本的模板可能就很难直接套用。这时候你需要去改写它的Iterator，甚至重写它的Main Loop。
这一改，就相当于是在CUTLASS的源码基础上做二次开发。这不仅要求你懂CUDA，还要求你懂CUTLASS那套极其复杂的C++ Template元编程技巧。很多时候，读懂CUTLASS源码的时间，比自己从头写一个针对特定Shape优化的Kernel还要长。
而且，开源库为了通用性，往往会有很多分支判断和复杂的参数计算，这些在编译成汇编后，可能会引入多余的指令。而针对特定模型手写的Kernel，可以把所有的参数都硬编码（Hardcode）进代码里，变成立即数。
这看似微小的差别，在核心循环（Main Loop）里执行几十亿次之后，积累下来的收益是惊人的。编译器很难大胆到把变量直接优化成立即数，因为通过编译器的静态分析很难确定这个变量在运行时真的不会变。但作为Kernel Engineer，我很清楚这个模型的Hidden Size就是4096，永远不会变，我直接写死，就能省下寄存器，省下指令发射槽。
一个具体的实战场景：MoE模型中的Gather/Scatter操作
让我们聊聊Mixture of Experts（MoE）模型，这是目前提升大模型参数量的关键技术，比如GPT和DeepSeek都在用。
MoE的核心逻辑是：根据输入的Token，动态选择一小部分专家（Experts，本质上是小的FFN层）进行计算。这就涉及到一个极其麻烦的数据搬运过程：要把Token分发给不同的专家，算完后再收集回来。
如果你用现成的算子拼凑，流程大概是：
计算路由得分（Softmax）。
根据得分做Argsort或者TopK。
根据索引做Index Select（访存极其不连续，带宽杀手）。
对每个专家做GEMM。
结果再做Scatter Add加回来。
这一套下来，中间产生的显存碎片和带宽浪费是巨大的。因为显存最怕这种随机读写（Random Access）。
这时候，Kernel Engineer怎么做？
他会设计一个专门的Fused MoE Kernel。利用Shared Memory作为中转站，在片上直接完成Token的重排（Permutation）。他会利用硬件的原子指令（Atomics）来高效处理Scatter Add。他会根据专家的数量和负载均衡情况，动态调整Block的发射策略。
这种级别的优化，现有的CUTLASS或者cuBLAS根本没有现成的接口。PyTorch Compile生成的代码在处理这种动态索引时，往往会退化成串行或者极其低效的Gather实现。
只有人，能根据数据的稀疏特性，设计出精妙的Block Mapping策略，让GPU这头吞吐量巨兽始终吃得饱饱的，不让显存控制器有喘息的机会。
关于未来的职业布局与“护城河”的再思考
前面提到了体系结构是护城河，这里我想再补充一点：
跨栈的理解能力
。
单纯的写CUDA代码，以后确实可能会被AI辅助编程工具（Copilot等）替代一部分。AI很擅长写这种模式固定的Boilerplate代码。
但是，AI很难理解
系统边界
。
一个优秀的GPU Engineer，不仅仅是在优化一个Kernel，而是在优化整个系统的Dataflow。
你需要判断，这个算子到底该不该融合？有时候融合了反而慢，因为寄存器压力太大导致Occupancy暴跌。
你需要判断，这个计算是放在GPU上做，还是放在CPU上做更合适？或者是不是应该利用网卡（NIC）的计算能力（In-Network Computing）？
你需要判断，当前模型的瓶颈是在计算（Compute Bound）、带宽（Memory Bound）还是在Kernel启动开销（Latency Bound）？
这种全局的System View，加上微观的Micro-architecture Insight，才是你立足不败之地的根本。
对于新人来说，不要被满屏的PTX汇编吓退。先从CUDA C写起，理解Warp Shuffle，理解Shared Memory Bank Conflict。然后学会看SASS，看看编译器到底把你写的C++变成了什么机器码。最后，去理解模型，去理解为什么Attention要是这个样子，为什么RMSNorm比LayerNorm好做优化。
只要大模型还在发展，只要人类还在追求更高的FLOPS/Watt，GPU Kernel Engineer就永远不会衰退。
